# Implementation Plans — mossferry experience improvements

Generated by the `improve` skill on 2026-07-18, planned against commit `88cd1f4` (v2.6.0).

These plans target the **overall experience**: the picker feels clunky (kill/rename tear
down and rebuild fzf), the client is round-trip-heavy (doctor hangs and is slow), and a
handful of correctness bugs can ship green. Each plan is self-contained for an executor with
zero context from the authoring session: read the plan fully, honor its STOP conditions, run
every verification command, and update your row in the table below when done.

> **This set was scoped by the maintainer** to four packages — the flagship picker redesign,
> connect/doctor UX, the correctness sweep, and foundation (tests + docs + a config knob).
> A CI GitHub Action (finding DX-02) was **deliberately excluded** (GitHub billing).

## ⚠️ Environment note — read before running any verification

**`shellcheck` is OPTIONAL and may be absent.** It is *not* installed in the authoring
environment and is not invoked by `tests/run.sh`. Wherever a plan lists `shellcheck <file>`
as a verification step or done-criterion, treat it as **skip-if-absent**:

```sh
command -v shellcheck >/dev/null && shellcheck <file>   # run only if present
```

A missing `shellcheck` is **never** a verification failure and **never** a STOP condition.
The always-available required gates are **`bash -n <file>`** (syntax) and **`bash tests/run.sh`**
(the suite). If you have `shellcheck`, honor the existing inline `# shellcheck disable=`
directives.

## Findings considered and rejected (so nobody re-audits them)

From the verified audit (37-agent adversarial pass). These were checked and deliberately **not** turned into plans:

- **CI GitHub Action (DX-02):** real gap (no CI, open-source intent) — **excluded by maintainer** for GitHub billing reasons, not because it lacks merit. Revisit when billing allows.
- **mapfile/flock "portability breach" (PORT-01 / CORRECTNESS-03):** **non-issue.** `bin/repo-session` is Linux-only by design; the "macOS bash 3.2" text is a function-local resolver docstring, not a script contract. No code change warranted (optionally one clarifying comment).
- **Second ssh handshake before mosh (PERF/connect):** **by design** — pre-validation runs over ssh so repo typos print locally ("errors are local"). Latency is addressed by `ControlMaster` in `~/.ssh/config`, which the design docs already prescribe.
- **load_config empty-value divergence (TECH-DEBT-07):** heavily overstated — the flagship variable has no client-side effect; only `FERRY_SERVER_TIMEOUT` empty matters (a dubious want). Optional one-char fix (`${VAR=default}`), not worth a plan.
- **attach-elsewhere marker / `attach -d` on picker enter (UX-04):** convention is fine; plain attach is intentional and `-d` is reserved for the grid-claim path. At most a cosmetic "elsewhere" glyph — not planned.
- **Validate ssh has no timeout/progress (PERF-03):** the hang belongs to user ssh config (`ConnectTimeout` prescribed in design docs); only a dim progress line would help — folded conceptually into future connect polish, not planned now.
- **cmd_update 3-4 serial ssh handshakes (PERF-04):** real latency, same class as doctor (plan 07). Deferred; apply the same `ControlMaster`/batched-`sh -s` approach as plan 07 when connect polish continues.
- **Version-drift warning piggyback on validate (connect UX-02):** genuine small win (the pre-validate round trip already sends `--client-version` but the answer is discarded) — deferred; land after plan 08 touches the same pre-validate path.
- **Launcher/kill key on wrong row flashes a rebuild (UX-05):** **by design** today, and it disappears for free once plan 03 moves the picker to persistent `--bind`s.
